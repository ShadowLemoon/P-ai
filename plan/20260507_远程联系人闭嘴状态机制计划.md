# 20260507_远程联系人闭嘴状态机制计划

## 一、目标

为远程联系人新增一层独立的“闭嘴机制”，插入到现有收信与在场判定之间，作为高优先级消息拦截器。

这次改动的核心不是重做原有“离场 / 在场 + 空闲 / 忙碌”状态机，而是在其前面增加一段可配置、可超时自动解除、可通过张嘴词解除的门禁链路。

目标行为：

1. 联系人可配置：
   - 闭嘴词
   - 张嘴词
   - 闭嘴时长（默认 10 分钟）
2. 默认预设：
   - 闭嘴词：`闭嘴`
   - 张嘴词：`张嘴`
   - 闭嘴时长：`600` 秒
3. 消息进入后的判定顺序固定为：
   - 收信权限判定
   - 闭嘴判定
   - 在场判定
   - 回复策略判定
   - 发信判定
4. 闭嘴期间直接拦截后续链路：
   - 不进入在场判定
   - 不进入回复策略判定
   - 不进入发信判定
5. 闭嘴解除有两种方式：
   - 命中张嘴词
   - 在闭嘴判定时发现超时

## 二、业务规则

## 2.1 联系人配置字段

在联系人配置上新增以下持久化字段：

1. `mute_keywords`
   - 闭嘴词列表
   - 字符串数组存储
   - 默认值：`["闭嘴"]`

2. `unmute_keywords`
   - 张嘴词列表
   - 字符串数组存储
   - 默认值：`["张嘴"]`

3. `mute_duration_seconds`
   - 闭嘴持续时间，单位秒
   - 默认值：`600`

输入规则：

1. UI 上仍用逗号分隔文本输入
2. 允许全角 `，` 与半角 `,`
3. 保存前自动做：
   - 分隔归一化
   - trim
   - 去空项
   - 去重
4. 后端同样做一层兜底归一化，避免前端漏网数据进入存储

## 2.2 运行时状态

新增一层独立运行时状态：

1. `mute_until: Option<String>`
   - 表示闭嘴截止时间
   - `None` 表示当前未闭嘴
   - `Some(timestamp)` 表示当前处于闭嘴期，直到该时间点

说明：

1. 这层状态独立于原有：
   - `presence_state: away | present`
   - `work_state: idle | busy`
   - `has_pending`
   - `last_success_reply_at`
2. 闭嘴不改写原有状态机字段
3. 闭嘴只影响“消息是否继续进入后续处理链路”

## 2.3 消息判定顺序

收到新远程消息后，按以下顺序处理：

### 1. 收信权限判定

若联系人或渠道不允许收信，则直接结束。

### 2. 闭嘴判定

这一步是新加的高优先级拦截器。

#### 2.3.2.1 先判超时

如果联系人当前存在 `mute_until`：

1. 若当前时间已超过 `mute_until`
   - 自动解除闭嘴
   - 本条消息继续通过，进入后续在场判定

2. 若尚未超时
   - 继续执行下面的命中规则

#### 2.3.2.2 命中闭嘴词

若本条消息命中闭嘴词：

1. 立即设置：
   - `mute_until = now + mute_duration_seconds`
2. 直接拦截本条消息
3. 后续：
   - 不进入在场判定
   - 不进入回复策略判定
   - 不进入发信判定

说明：

1. 闭嘴词优先级高于普通消息放行
2. 闭嘴词本身也是一条“触发闭嘴并拦截”的指令消息

#### 2.3.2.3 当前处于闭嘴且命中张嘴词

若当前未超时但仍处于闭嘴期，同时本条消息命中张嘴词：

1. 立即解除闭嘴：
   - `mute_until = None`
2. 本条消息继续放行
3. 后续按正常链路继续：
   - 进入在场判定
   - 进入回复策略判定
   - 进入发信判定

注意：

1. 这里的“放行”不等于“强制入场”
2. 后续完全按原有在场规则正常判断

#### 2.3.2.4 当前处于闭嘴但未命中张嘴词

若当前仍在闭嘴期，且本条消息未命中张嘴词：

1. 直接拦截
2. 不进入后续链路

### 3. 在场判定

只有通过闭嘴判定后，才会进入原有 away/present 逻辑。

### 4. 回复策略判定

保持现有 `always_reply / smart_judge` 语义不变。

### 5. 发信判定

保持现有 send / no_reply / async send 行为不变。

## 三、冲突与优先级

本次第一版按以下优先级处理：

1. 收信权限优先
2. 闭嘴超时自动解除优先于“继续拦截”
3. 闭嘴词优先于张嘴词
4. 张嘴词只负责“解除闭嘴并放行”
5. 放行后仍由原状态机决定是否在场、是否回复

说明：

1. 若同一条消息同时命中闭嘴词与张嘴词：
   - 按“闭嘴词优先”
   - 结果为重新进入闭嘴并拦截
2. 第一版不做更复杂的多轮口令语义判断

## 十、风险与边界

1. 闭嘴状态目前计划放运行时内存，不跨重启保留
   - 应用重启后闭嘴自动失效
   - 第一版暂不持久化

2. 若同一条消息同时命中闭嘴词与张嘴词
   - 第一版按闭嘴优先

3. 第一版只处理“纯包含式关键词命中”
   - 不做分词
   - 不做正则
   - 不做更复杂上下文指令理解

4. 第一版默认不新增联系人列表实时“闭嘴中”徽标
   - 若后续需要可单独补 UI 可视化

## 十一、完成标准

满足以下条件即可认为本次完成：

1. 联系人设置中可编辑闭嘴词、张嘴词、闭嘴时长
2. 默认预设为：
   - 闭嘴
   - 张嘴
   - 10 分钟
3. 入站消息按“收信权限 -> 闭嘴判定 -> 在场判定 -> 回复策略 -> 发信”顺序执行
4. 闭嘴期间普通消息被拦截
5. 张嘴词和超时都能解除闭嘴并放行
6. 放行后继续走原有状态机，不强制入场
7. 必要测试通过，且无明显无关改动扩散
