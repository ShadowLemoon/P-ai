# 20260508 计划文件化重构计划

## 1. 背景

当前 `plan` 机制把计划正文直接放在工具参数和消息元数据里，这会带来几个明显问题：

1. LLM 每次提交或修改计划时，都要重复输出整份计划正文。
2. 如果只想修改一个小段，也必须整段重发，无法自然地走补丁式编辑。
3. 前端展示、执行中计划记录、提示词注入都在重复存储同一份计划文本，容易漂移。
4. 用户确认的是“这一段临时文本”，而不是“一个稳定存在的计划文件”。

这次重构的目标不是继续优化 `context` 文本，而是把计划改成“文件为中心”的模型。

## 2. 目标

把计划协议重构为：

1. LLM 先在文件系统中写出计划文件。
2. `plan` 工具只提交计划文件路径，而不是整段计划正文。
3. 前端渲染直接读取该计划文件。
4. 执行中计划记录只保存路径等引用信息，不再复制整段正文。
5. 提示词回灌时只注入计划路径，让模型按需读取或补丁修改。

## 3. 核心方案

### 3.1 默认计划目录约定

计划文件默认应输出到：

1. 优先：`会话工作目录/.pai/plan/`
2. 回退：`自我目录/.pai/plan/`

说明：

1. 这是写入约定，主要通过 `plan` 工具规则和相关提示词告诉 LLM 应该把计划写到哪里。
2. 如果当前会话存在 `会话工作目录（Session Working Directory）`，则计划应优先写入该目录下的 `.pai/plan/`。
3. 如果当前会话没有工作目录，则回退到 `自我目录（Self Directory）/.pai/plan/`。
4. 后端不负责替 LLM 决定默认文件名或主动生成计划文件，只负责在 `plan` 提交时校验路径是否符合约定。

### 3.2 `plan` 工具改为路径协议

`plan` 工具不再以整段正文作为计划本体，而改为引用计划文件。

建议协议：

1. `plan.present`
   - 必填：`path`
   - 作用：提交这份计划文件，供用户确认与后续执行。
2. `plan.complete`
   - 必填：`path`
   - 作用：按该路径对应的计划进行完成登记。

完成后的汇报不再通过 `plan.complete` 参数承载，而是由模型在工具调用后继续自然回复给用户。

### 3.3 计划文件格式

计划文件先采用 Markdown：

1. 后缀固定为 `.md`
2. 内容允许 LLM 直接整写，也允许后续使用 `apply_patch` 局部修改

计划文件命名建议：

1. `plan.md`
2. 或 `plan_<时间戳>.md`

本轮实现时应优先保证：

1. 路径稳定可读
2. 前端可直接渲染
3. LLM 可以方便地对同一文件做补丁修改

是否采用单文件复用还是每次 `present` 新建文件，本轮需要在实现时定死策略。

### 3.4 执行中计划记录

当前 `active_plans.jsonl` 存的是正文 `context`，本轮改为存引用信息。

至少应保存：

1. `plan_id`
2. `source_message_id`
3. `status`
4. `path`
5. `created_at`
6. `completed_at`
7. `completion_text`

不再把整段计划正文写进执行中计划记录。

### 3.5 用户确认链路

当前用户确认逻辑是从 `planCard.context` 读取计划文本，本轮改为：

1. `plan.present` 消息里携带 `path`
2. 用户确认时后端读取该 `path`
3. 将其登记为执行中计划

也就是说：

1. 用户确认的是一份文件
2. 系统继续执行时引用的是同一份文件

### 3.6 提示词注入链路

当前活跃计划注入会把整段正文拼进 `<active_plans>`，本轮改为注入路径引用。

目标效果：

1. 提示词里只告诉模型“当前执行计划位于哪个路径”
2. 模型如需查看全文，可自行读取
3. 模型如需调整某一小段，可直接补丁修改对应文件

这样可以避免每轮提示词都重复搬运整段计划。

### 3.7 前端渲染链路

前端不再依赖消息元数据里的整段计划正文，而改为：

1. 读取消息中的计划路径
2. 请求后端读取该文件
3. 直接渲染 Markdown

这样前端展示与文件真源保持一致。

## 4. 关键决策点

### 4.1 计划文件是否允许覆写

这里有两种方向：

1. 同一路径持续修改同一份计划
2. 每次 `present` 生成一份新计划文件

本次讨论里，用户更关注“LLM 能对现有计划做补丁修改”，因此当前优先方向是：

1. 同一路径持续维护同一份计划文件

但需要同步考虑：

1. 历史消息是否应显示当时版本还是当前最新版本
2. 前端渲染是否接受“旧消息看到的是新文件内容”

如果后续认为历史快照也必须稳定，再补版本化方案。

### 4.2 `plan.present` 是否保留 `context`

本轮建议：

1. `plan.present` 不再传计划正文
2. 如需说明性文字，可由普通助手消息承担
3. `plan` 工具只负责提交计划文件引用

这样工具语义最干净。

### 4.3 路径权限边界

`plan` 工具提交的路径不能放开成任意绝对路径。

必须限制为：

1. 当前 `会话工作目录/.pai/plan/` 下
2. 或当前 `自我目录/.pai/plan/` 下

否则会出现：

1. 计划文件写到用户项目任意位置
2. 前端回放找不到
3. 历史计划与会话生命周期脱钩

## 5. 非目标

本轮不做：

1. 计划文件版本历史浏览器
2. 计划 diff 可视化
3. 多计划树状管理界面
4. 计划文件自动摘要
5. 把所有任务 / todo 也一起改成文件协议

## 7. 风险点

### 7.1 历史消息与当前文件内容可能不一致

风险：

1. 如果同一路径持续被修改
2. 老消息重新打开时，看到的可能是最新版本，而不是当时版本

处理：

1. 本轮先接受“计划文件是活文档”语义
2. 若后续需要历史快照，再引入版本化

### 7.2 路径失效

风险：

1. 如果计划文件被删除或移动
2. 前端渲染和执行中计划读取都会失败

处理：

1. 所有读取失败都必须降级成可读提示
2. 禁止因为计划文件缺失而打断主业务

### 7.3 LLM 写入位置不稳定

风险：

1. 如果提示词没有明确默认目录
2. LLM 可能把计划写到错误位置

处理：

1. 在 `plan` 工具规则和相关提示词中明确默认计划目录
2. `plan.present` 对路径做严格校验，不符合目录规则就拒绝登记

## 8. 验证

### 8.1 Rust

1. `cargo check`
2. `plan.present(path=...)` 只能接受允许目录内的计划文件
3. 无会话工作目录时，能正确回退到 `自我目录/.pai/plan/`
4. `plan.complete(path, context)` 不再终止工具循环
5. 执行中计划记录只保存路径，不再保存整段正文
6. 活跃计划提示词注入改为路径引用
7. 计划文件缺失时返回可读降级，不打断业务

### 8.2 Frontend

1. 计划卡片可根据路径读取并渲染
2. 用户确认后仍能继续按计划执行
3. 历史消息回放时若文件丢失，能给出明确提示

## 9. 实施顺序

1. 重写 `plan` 工具协议，从正文模式切到路径模式。
2. 重构执行中计划存储，把 `context` 改为 `path` 引用。
3. 调整确认继续执行链路，改为读取计划文件路径。
4. 调整活跃计划提示词注入，改为只注入路径。
5. 接入前端计划文件读取与渲染。
6. 补齐降级处理和最小必要测试。

## 10. 预期结果

完成后，计划将从“聊天里反复输出的大段文本”变成“可持续编辑的计划文件”：

1. LLM 只需维护一份计划文件。
2. 修改局部内容时只需补丁编辑。
3. 前端、提示词、执行状态都共享同一份计划源。
4. `plan` 工具语义回归为“提交计划引用”，而不是“搬运整段正文”。
