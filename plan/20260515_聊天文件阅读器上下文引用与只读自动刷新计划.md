# 聊天文件阅读器上下文引用与只读自动刷新计划

## 背景

聊天右栏已经整合文件阅读器，但阅读器和对话输入之间缺少自然的“我正在看/我选中了这段”的上下文桥接。用户希望在 agent 驱动的 vibe coding GUI 中，文件片段可以被显式引用给模型，同时阅读器只做查看，不承担编辑器职责。

此外，文件可能被外部编辑器、agent 或工具修改。阅读器作为只读查看器，需要让当前正在看的文件和目录保持新鲜，但不做编辑冲突处理、不做全项目 watcher，也不为后台 tab 消耗资源。

## 目标

- 文件阅读器打开文件后，输入区始终有一份“当前可视文件片段”候选上下文。
- 用户选中文本时生成二级候选；当二级候选与一级候选同文件时，以二级候选为准。
- 用户取消选区时清空二级候选，恢复一级候选展示。
- 候选不会自动发送，只有用户点击 `+` 后才作为附件快照附加。
- 附加后的内容与候选解耦，后续滚动或重新选择不会修改已附加快照。
- 发送给模型的提示词只表达显式引用：`用户引用了文件片段：path:start-end`。
- 附加正文超过 2000 字符时，不发送正文，只提示引用总字符数。
- 阅读器自动刷新只覆盖当前 active tab 文件和可见目录节点，保持完全只读。

## 非目标

- 不做文件编辑器能力。
- 不处理 dirty 状态、冲突合并、保存提示。
- 不递归监听整个工作区。
- 不监听后台 tab。
- 不做代码/markdown 内容的局部增量渲染引擎。
- 不把“用户正在阅读”作为发送给模型的提示词，避免误导。

## 交互规则

### 上下文候选

- 一级候选：当前 active 文件的可视范围，单例。
- 二级候选：当前文本选区，单例。
- 派生展示规则：
  - 没有二级候选时，展示一级候选。
  - 二级候选与一级候选同文件时，展示二级候选。
  - 二级候选与一级候选不同文件时，展示一级候选。
  - 取消选择时，只清空二级候选。

### 附加规则

- 点击候选 chip 的 `+` 后，复制当前候选为已附加快照。
- 同文件同行号范围已经附加时，隐藏候选 chip。
- 同文件但不同行号范围时，候选仍可再次附加。
- 点击附件按钮不抢占输入框焦点，点击后恢复输入光标。

### 行号规则

- code/raw 视图：如果选中文本在源文件中唯一出现，使用精确行号。
- code/raw 视图：如果选中文本重复出现，退回可视范围估算行号，避免错误定位到第一个匹配。
- markdown 渲染视图：不做不稳定的源文本精确匹配，统一使用可视范围估算行号。

## 技术方案

### 前端

- `FileReaderPanel.vue`
  - 捕获 active 文件可视范围，生成一级候选。
  - 捕获文本选区，生成二级候选。
  - 取消选区时发出清空二级候选事件。
  - 生成统一 `IdeContextReferenceItem`，附带 `textBlock` 快照文本。
  - 通过滚动、切换 tab、文件加载完成刷新候选。
  - 计算只读自动刷新目标：active 文件、目录根、目录树当前可见 entry。

- `ChatView.vue`
  - 持有一级/二级候选状态。
  - 按派生规则向 Composer 提供最终候选。
  - 会话切换时清空候选。

- `ChatComposerPanel.vue`
  - 上下文 chip 按文件路径与行号范围合并，不按工作区分组。
  - 已附加同范围时隐藏候选。
  - `mousedown.prevent` 避免附件按钮抢焦点，切换后恢复输入框焦点。

### 后端

- `desktop_tools.rs`
  - 新增 `update_file_reader_watch_targets` 命令。
  - 不引入 watcher 依赖，使用轻量轮询指纹。
  - 每个 session 保存当前监听目标与指纹。
  - 指纹包含存在性、是否目录、文件长度、修改时间、目录签名。
  - 目录签名最多扫描 100 个 entry。
  - 发现变化后通过 `easy-call:file-reader-watch-changed` 推送给前端。

### 自动刷新边界

- 文件事件只刷新当前 active 文件。
- 目录事件只刷新对应目录节点，不刷新整个目录树。
- 可见文件 entry 变化不触发目录树刷新。
- 刷新失败复用现有错误展示。

## 验证

- `pnpm typecheck`
- `cargo check`
- `git diff --check`

## 风险与后续

- 轮询指纹不是实时 watcher，存在最多约 700ms 延迟，但对只读阅读器足够稳定。
- markdown 渲染态行号是估算值，不承诺精确源行。
- 后续如果需要 VS Code 级局部渲染，应考虑接入 Monaco/CodeMirror 只读模型，而不是手写增量 DOM 更新。
