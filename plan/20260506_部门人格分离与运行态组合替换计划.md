# 20260506 部门人格分离与运行态组合替换计划

## 一、目标摘要

本计划用于完成历史 `agent` 语义拆分，但**本轮只写计划，不进入实现**。

本次重构的核心结论已经确定：

- 程序内部真实身份模型永远是两个独立字段：
  - `department_id`
  - `persona_id`
- `runtime agent` 不是系统主身份，也不是持久化主键。
- `runtime agent` 只是给用户选择、展示、日志说明、`@` 列表使用的临时组合视图。
- 所有运行链路、持久化链路、接口链路，最终都应回到“部门 / 人格分离”的结构。

本次计划不是“全系统切成 runtime agent 主键”，而是：

1. 清理旧 `agent` 混合语义。
2. 把所有真实业务数据还原为 `department_id + persona_id`。
3. 仅在 UI 和局部运行态中保留“部门人格组合项”作为便捷视图。

---

## 二、最终语义定义

### 1. 会话层

- `conversation.department_id`
  - 表示当前会话的**主激活部门**。
- `conversation.persona_id`
  - 表示当前会话的**主激活人格**。
- 这两个字段共同描述“当前谁在主持这个会话”。
- 后续如果用户主动更换主持部门或主持人格，只改会话头，不改历史消息。

### 2. 消息层

- `message.speaker_persona_id`
  - 表示“这条消息是谁说的”。
- 消息层**不存 `speaker_department_id`**。
- 原因：
  - 部门只是会话当前上下文，不是每条消息的永久属性。
  - 发言者可以说话，但不必拥有一个随消息落盘的部门身份。
  - 否则会制造“消息部门”和“会话部门”双重真相。

### 3. 联系人 / 委托 / 调度上下文

- 凡是需要描述“当前主处理位 / 当前路由目标 / 当前执行者”的地方，统一存：
  - `department_id`
  - `persona_id`
- 不再持久化单独的运行态组合 ID。

### 4. runtime agent

- `runtime agent` 只是一种临时组合视图。
- 用途仅限：
  - UI 选择项
  - `@` 候选列表
  - 调试日志展示
  - 秘书提示词中的“当前是谁的秘书”
  - 局部运行态缓存
- 它不是：
  - 持久化字段主键
  - 前后端请求主键
  - 内部统一身份主键

## 三、需要落地的结构调整

### 1. 持久化字段重命名

- `Conversation.agent_id -> persona_id`
- `ChatMessage.speaker_agent_id -> speaker_persona_id`

同时保留兼容读取逻辑：

- 启动加载旧数据时，允许读取旧字段名。
- 装载后自动映射到新字段。
- 新数据写回时只写新字段名。

### 2. 会话相关结构统一

以下所有“当前主处理者”语义，统一改为 `department_id + persona_id`：

- 会话主记录
- 会话概览
- 未归档会话摘要
- 归档摘要
- 快照与导出结构
- 会话运行时上下文中需要长期保存的部分

### 3. 消息相关结构统一

以下所有“谁说了这句话”语义，统一改为 `speaker_persona_id`：

- 会话消息
- 归档消息
- 快照 API 返回消息
- 渠道日志里引用消息发言者的展示数据来源

### 4. 联系人绑定统一

当前远程联系人只绑定 `bound_department_id`，这不再足够。

需要改为同时持久化：

- `bound_department_id`
- `bound_persona_id`

含义是：

- 联系人默认由哪个部门的哪个人格接待。
- 这是联系人自己的绑定配置，不依赖运行时临时推断。

### 5. 委托链路统一

委托记录与委托线程不再存模糊 `source_agent_id / target_agent_id` 语义，而是显式拆成：

- `source_department_id`
- `source_persona_id`
- `target_department_id`
- `target_persona_id`

运行时需要展示组合身份时，再临时拼出视图项。

### 6. 运行时上下文统一

当前运行时上下文里的这类字段都需要拆开：

- `executor_agent_id`
- `source_agent_id`
- `target_agent_id`
- `SessionSelector.agent_id`

统一改造成“部门 / 人格分离传递”：

- `executor_department_id + executor_persona_id`
- `source_department_id + source_persona_id`
- `target_department_id + target_persona_id`
- `SessionSelector.department_id + persona_id`

如果某处只是“消息发言者”，则只保留 `speaker_persona_id`，不补部门。

### 7. 错误遗留字段清理

`assistantDepartmentAgentId` 定义错误，后续应完全退出正式模型。

处理原则：

- 启动迁移时兼容读取。
- 新结构中不再继续使用它表达主助理。
- 任何主激活人格、主激活部门，都必须来自真实会话上下文、联系人绑定或明确配置字段，而不是这个兜底字段。

## 四、迁移策略

### 1. 允许自动修正的内容

迁移层只负责“字段语义拆分”和“字段名升级”，不负责擅自改变业务归属。

允许自动处理：

- 旧 `agent_id` 自动映射为 `persona_id`
- 旧 `speaker_agent_id` 自动映射为 `speaker_persona_id`
- 旧结构里同义字段统一归并到新字段
- 旧 `assistantDepartmentAgentId` 自动忽略或转入兼容读取逻辑

### 2. 不允许自动改变的内容

迁移层禁止做以下事：

- 因为当前部门配置变化，自动替换历史会话的人格
- 因为当前部门不再挂这个人格，自动改绑到另一个人格
- 因为委托目标当前不存在，偷偷改写历史委托记录

也就是说：

- 历史数据表达的是历史事实
- 迁移层只能改字段形状，不能改业务含义

### 3. 与当前部门配置的关系

如果旧数据原本能落盘，就说明它当时是合法事实。

因此迁移后的持久化数据：

- 不依赖“当前部门树是否仍然这样绑定”来决定是否成立
- 不需要通过当前配置重新推导一遍“真正身份”
- 只要原字段存在，就原样保留其业务语义

运行期如果要检查“当前还能不能继续调度”，那是运行时逻辑，不是迁移逻辑。

## 五、功能链路改造范围

### 1. 前端

- `@` 候选列表改为基于“部门 + 人格”的组合项展示
- 会话列表与聊天头部展示主激活部门、主激活人格
- 消息项只根据 `speaker_persona_id` 渲染发言者
- 头像选择与可召唤判断复用同一套部门/人格组合来源
- 设置页与联系人设置页都明确编辑“部门 + 人格”绑定，而不是旧 `agent`

### 2. 聊天调度

- 会话调度入口统一读取 `conversation.department_id + conversation.persona_id`
- 秘书判断读取当前会话主激活位，不读取历史遗留兜底字段
- 联系人调度读取联系人绑定的 `bound_department_id + bound_persona_id`
- 子任务和委托调度统一传递部门/人格对，不再在内部把它压扁成单个 `agent_id`

### 3. 工具与提示词

- 工具运行上下文中的执行者字段改为部门/人格分离
- reload、日志、调试输出如果需要人类可读身份，可额外附加组合展示文本
- LLM 提示词里如果需要表达“当前是谁在处理”，应显式输出：
  - 当前部门
  - 当前人格
  - 联系人/群友等对端对象

### 4. 归档与快照

- 归档记录中的主持人改为 `department_id + persona_id`
- 历史消息发言人只保留 `speaker_persona_id`
- 快照 API、导出、归档窗口渲染全部同步升级，避免继续复用旧字段名

### 5. 记忆例外

记忆系统保持现有产品语义：

- 记忆继续按 `persona_id` 共享
- 不按部门拆分
- 不按 runtime 组合拆分

因此所有进入记忆层的调用都必须先取人格，再做记忆归属。

`owner_agent_id` 后续需要迁移为更准确的人格语义命名，或至少在适配层中明确其真实含义。

## 六、建议实施顺序

### 阶段一：数据模型与类型收口

- 后端核心结构增加/切换 `persona_id`
- 前端类型同步切换
- 保留旧字段兼容读取
- 新写入只写新字段

### 阶段二：会话与消息链路

- 会话创建、读取、更新全部改走 `department_id + persona_id`
- 消息发言者链路改走 `speaker_persona_id`
- 快照、归档、列表、消息渲染同步收口

### 阶段三：联系人与秘书

- 联系人绑定补齐 `bound_persona_id`
- 联系人激活、秘书判断、入场策略统一读取“部门 + 人格”
- 渠道日志输出当前部门、当前人格、联系人信息

### 阶段四：委托与运行时上下文

- 委托记录、委托线程、异步委托入口拆分 `source/target department/persona`
- 所有 runtime context 字段完成去 `agent_id` 化
- 工具运行上下文同步升级

### 阶段五：遗留字段下线

- 清理 `assistantDepartmentAgentId` 正式用途
- 清理剩余错误命名
- 收缩兼容层，只保留必要历史读取

## 七、验证要求

### 1. 会话与消息语义

- 修改会话主持人格后，历史消息发言人不变
- 修改会话主持部门后，历史消息发言人不变
- 消息渲染只依赖 `speaker_persona_id`

### 2. 同人格多部门

- 同一个人格可以同时出现在多个部门的候选列表中
- UI 选择时能区分“哪个部门的哪个人格”
- 但消息历史里说话者仍只显示对应人格事实

### 3. 联系人

- 每个联系人都能独立绑定部门 + 人格
- 联系人入场、秘书判断、实际调度三者使用同一套绑定语义
- 联系人配置修改后，只影响后续轮次，不篡改历史消息

### 4. 委托

- 委托来源与目标都能明确记录部门 + 人格
- 委托结果消息的发言者仍按真实人格写入
- 同人格跨部门委托时不再因旧 `agent_id` 混用而歧义

### 5. 迁移

- 旧数据能被正确读取
- 新数据只写新字段
- 不因当前部门树变化而重写历史会话语义

## 八、本轮不做的事

- 本轮不实现代码改造
- 本轮不提交迁移代码
- 本轮不清理所有旧测试
- 本轮不改动记忆产品语义

本计划文档只作为下一轮正式重构的执行依据。
