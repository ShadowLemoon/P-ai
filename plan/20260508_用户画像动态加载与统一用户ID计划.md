# 用户画像动态加载与统一用户 ID 计划

## 1. 背景与问题

当前用户画像链路存在两个混在一起的问题：

- 画像加载仍偏静态快照：会话创建、压缩、归档等路径会提前组装并保存 `conversation.user_profile_snapshot`，后续提示词再读取这份快照。
- 画像判定口径被错误地设计成内部结构标签：把“是否画像、归属用户、属性类型”编码进类似机器字段的标签，会污染记忆标签的搜索职责。

用户画像仍然属于长期记忆系统。它不应该脱离 `memoryActions`，也不应该使用另一套隐藏结构。画像记忆必须仍然是一条普通记忆，只是它的 `tags` 遵循更严格的搜索锚点规则。

## 2. 设计原则

- `tags` 的本意是搜索锚点，不允许塞伪结构字段。
- 用户画像记忆仍然通过 `memoryActions` 新增、修正、合并，不新增单独的画像输出结构。
- 用户画像属于“人”，不属于群、会话、联系人路由对象。
- 记忆层不理解 OneBot / 微信 / 飞书 / 钉钉等平台字段，只消费注入层给出的统一用户 ID。
- 用户 ID 是不透明字符串，必须原样保存、原样匹配，不转数字、不截断、不改写大小写。
- 画像属性标签必须使用固定中文标签，不能改成英文枚举。

## 3. 用户画像标签规则

### 3.1 固定画像属性标签

用户画像属性标签固定为 6 个：

- `用户别名`
- `事实属性`
- `技能树`
- `关系图谱`
- `活跃项目`
- `用户要求`

不得新增英文替代标签，不得把这些标签改写成代码字段风格。

### 3.2 用户画像记忆的 tags 结构

涉及具体用户时，用户画像记忆的 `tags` 必须严格使用以下核心结构：

- 用户昵称
- 用户 ID
- 画像属性标签

例如：

```json
{
  "memoryType": "knowledge",
  "judgment": "用户昵称（用户ID）要求 AI 回答时必须优先使用中文，不要擅自切换英文术语。",
  "reasoning": "用户明确纠正过回答中擅自使用英文标签和英文示例的问题。",
  "tags": ["用户昵称", "用户ID", "用户要求"]
}
```

`tags` 中不要把大量细碎对象都塞进去；具体对象、项目名、关系对象、要求内容写入 `judgment` 和 `reasoning`。

### 3.3 用户画像判定规则

只有同时满足以下条件的记忆，才视为用户画像记忆：

- `tags` 中存在目标用户 ID。
- `tags` 中存在 6 个固定画像属性标签之一。

用户昵称只用于可读性和辅助召回，不作为用户画像判定的唯一依据。

普通带人名的记忆不应被误判为用户画像。普通带“项目”“关系”等词的记忆也不应被误判为用户画像。

### 3.4 用户 ID 锚点

- 本地用户使用统一的本地用户 ID。
- 远程 IM 用户使用注入层提供的真实发言人 ID。
- 群消息中，画像归属当前发言人，不归属群。
- `judgment` 中应同时写出用户昵称和用户 ID，优先格式：`昵称（用户ID）`。
- 不要只用昵称作为用户标识，因为昵称可能变化或重复。

## 4. 六类画像属性含义

### 4.1 用户别名

记录用户稳定使用或明确指定的昵称、称呼、代号、常用名。

此类记忆的画像属性标签必须使用 `用户别名`。

### 4.2 事实属性

记录用户主动透露、长期稳定、未来对话可复用的低敏事实属性。

事实属性应保持具体、可验证，不做性格推断。禁止记录身份证、手机号、详细住址、密码、密钥等敏感信息。

### 4.3 技能树

记录用户已经掌握、正在学习或经常使用的技能、工具、语言、框架。

技能树应优先记录稳定能力，不记录一次性尝试。

### 4.4 关系图谱

记录稳定的关系边，语义上保持“用户 - 关系 - 对象”清晰。

关系对象可以是具体人、群体、AI、人设、组织、项目、作品、地点或长期任务。关系对象写入 `judgment`，不要额外扩展大量 `tags`。

### 4.5 活跃项目

记录用户正在持续推进的项目、插件、作品、研究、长期任务。

活跃项目记忆必须回答：谁在推进、项目是什么、当前稳定目标或方向是什么。不记录一次性临时待办，除非它属于长期项目的当前阶段。

### 4.6 用户要求

记录用户明确提出、未来对话需要持续遵守的稳定要求、边界、禁忌或工作方式约束。

用户要求不同于一次性指令。只有当要求具有后续复用价值，或用户明确表达“以后都要这样做 / 不要这样做 / 这个规则不能改”时，才生成 `用户要求` 画像。

## 5. 禁止画像内容

禁止生成以下人物画像记忆：

- 身份证号、护照号、社保号、银行卡号、手机号、详细住址等高敏身份或财务信息。
- 密码、验证码、密钥、token、cookie、私密链接等认证信息。
- 单次情绪、一次性口误、玩笑话、不可复用的临时状态。
- 对用户的贬损性、武断性格判断、无依据推测。
- 只对当前任务有效的临时待办。

## 6. 画像冲突与修正

- 当新信息与旧用户画像冲突时，优先修正画像，不要长期并存互相矛盾的画像记忆。
- 发现同一用户在同一画像维度下有重复、过时或冲突记忆时：
  - 相似收敛优先使用 `merge`。
  - 明确纠正旧事实优先使用 `update`。
- 若用户明确纠正旧画像，优先使用 `update`，不能仅 `create` 一条新画像。
- 明确纠正旧画像时，`sourceMemoryIds` 只写 1 条旧画像 ID。
- 合并或更新后的画像，`tags` 仍必须严格保持：用户昵称 + 用户 ID + 画像属性标签。

## 7. 统一用户 ID 职责

### 7.1 注入层

- 负责识别当前轮次发言人。
- 负责提供当前轮次的用户昵称与用户 ID。
- 负责区分“会话联系人 ID”“群 ID”“路由 ID”和“真实发言人用户 ID”。

### 7.2 记忆层

- 只消费注入层给出的用户 ID。
- 只按用户 ID 与固定画像属性标签查询画像记忆。
- 不关心平台原始字段名，不猜测平台身份语义。

### 7.3 会话层

- 可以保存当前用户画像快照，但快照必须绑定明确用户 ID。
- 用户 ID 切换时，不能复用上一人的画像快照。
- 群会话可以挂在群上，但画像必须挂在人上。

## 8. 用户画像查询与组装

查找某个用户的画像时，使用以下条件：

- `tags` 包含目标用户 ID。
- `tags` 包含 6 个固定画像属性标签之一。

查询结果组装为用户画像快照或画像板时，应保留：

- 记忆 ID。
- 记忆类型。
- 判断内容。
- 理由。
- 原始标签。

画像快照给主对话 LLM 使用，帮助压缩后延续用户背景和长期要求。画像板给整理 LLM 使用，帮助合并、修正、去除重复或过时画像。

## 9. 上下文注入规则

- 默认只注入当前发言人的画像。
- 不注入其他用户画像，除非当前轮次确实涉及多名发言人且需要区分身份。
- 单轮最多注入 3 个画像块。
- 秘书链路不允许看到任何记忆：
  - 不注入长期记忆。
  - 不注入用户画像。
  - 不注入 recall memory block。
  - 只允许看到秘书判断当前轮是否应答所需的最小上下文。
- 如果拿不到画像，直接不注入，不伪造，不拿别人的画像兜底。

## 10. 本地与远程边界

- 本地普通聊天是否注入本地用户画像，由具体链路边界决定；但只要注入，必须按本地用户 ID 查询。
- 本地摘要链路如需用户画像，只能注入本地用户画像。
- 远程 IM 普通聊天可按当前发言人临时注入画像。
- 远程会话摘要链路是否注入画像必须单独评估，不能默认把联系人、群或路由对象当用户画像主键。

## 11. 迁移策略

- 不再以旧画像链接表作为主读取入口。
- 旧画像数据若需要迁移，必须转为符合“用户昵称 + 用户 ID + 画像属性标签”的 tags 结构。
- 迁移时不能生成机器字段式标签。
- 迁移后主读取路径按用户 ID 与 6 个固定画像属性标签查询。

## 12. 预期改动范围

### 12.1 提示词与记忆整理规则

- 用户画像特别规则改为本文 6 类中文画像属性标签。
- 示例必须使用中文画像属性标签。
- 删除机器字段式画像标签示例。

### 12.2 用户画像识别与查询

- 画像识别改为：目标用户 ID + 固定画像属性标签。
- 画像查询改为 SQL 级过滤目标用户 ID 与固定画像属性标签。
- 查询时保持用户 ID 原样匹配。

### 12.3 用户画像组装

- 画像快照与画像板必须来自同一套画像判定规则。
- 组装结果用于不同消费者时可以有不同展示形态，但底层画像集合必须一致。

### 12.4 测试

- 覆盖 6 个画像属性标签。
- 覆盖复杂用户 ID 精确匹配。
- 覆盖昵称重复但用户 ID 不同的隔离场景。
- 覆盖普通带人名记忆不被误判为画像。
- 覆盖机器字段式旧标签不再作为新画像规范输出。

## 13. 分步实施

### 13.1 第一步：修正提示词规则

- 将用户画像特别规则改为 6 个中文画像属性标签。
- 新增 `用户要求`。
- 删除英文属性标签和机器字段式标签示例。

### 13.2 第二步：修正画像判定

- 定义固定画像属性标签集合。
- 按“用户 ID + 固定画像属性标签”判断画像。
- 保证用户 ID 原样匹配。

### 13.3 第三步：修正画像查询与组装

- 按目标用户 ID 快速查询画像。
- 组装当前用户画像快照与完整画像板。
- 确保主对话 LLM 与整理 LLM 看到的画像来自同一套规则。

### 13.4 第四步：迁移或兼容旧画像数据

- 对旧画像数据制定一次性迁移方案。
- 迁移后不再新增旧机器字段式标签。

### 13.5 第五步：验证远程 IM 身份边界

- 群消息按发言人用户 ID 加载画像。
- 联系人路由 ID、群 ID、会话 ID 不作为画像用户 ID。

## 14. 验收标准

- 提示词中固定画像属性标签为：`用户别名`、`事实属性`、`技能树`、`关系图谱`、`活跃项目`、`用户要求`。
- 新生成用户画像的 `tags` 严格为用户昵称、用户 ID、画像属性标签这类搜索锚点。
- 不再生成机器字段式画像标签。
- 能按目标用户 ID 查出该用户画像。
- 昵称相同但用户 ID 不同的用户不会串画像。
- 群消息按发言人加载画像，而不是按群加载画像。
- 用户明确提出的长期边界或要求可生成 `用户要求` 画像。
- 普通记忆仍可正常按 tags 搜索，画像标签不会污染搜索语义。
